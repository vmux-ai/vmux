# In-Code API Docs → vmux.ai/docs/api

**Date:** 2026-06-27
**Status:** Design (approved direction, pending spec review)
**Branch:** `feat/api-docs-site`

## Summary

Write API documentation as rustdoc comments (`//!`, `///`) directly in the Rust
source of all workspace crates, generate a structured model from it with rustdoc's
JSON output, and render that model as a **full API reference** under
`vmux.ai/docs/api` — styled in the existing vmux docs UI, not docs.rs HTML.

Nothing is published to crates.io or docs.rs. The source of truth is the code; the
website is the only rendering target.

This is two efforts in one spec:

- **Part A — Pipeline + UI:** the `vmux_docs` generator, a stable serde model, the
  website renderer/routes, Makefile + CI integration, and seed docs for every crate.
- **Part B — Full authoring:** a per-crate content pass writing `///` across the
  entire public API of all 19 crates, with conventions, a definition of done, and an
  ordering. Part B is mostly a content checklist that cannot be designed up front;
  Part A is the engineered system that makes it visible.

## Goals

- Documentation lives next to the code and is generated by rustdoc (no hand-mirrored
  prose API docs).
- Full public-API reference for **all 19 workspace crates** (`crates/*`).
- Rendered in current vmux style (the `/docs` look: `DocsLayout`, sidebar, TOC,
  syntect code), reusing the existing Markdown renderer for doc-comment bodies.
- The website deploy stays light: it never compiles the heavy (bevy/CEF) crates.

## Non-Goals

- No publishing to crates.io / docs.rs.
- No reskinning of `cargo doc` HTML (rejected — fragile, won't match vmux).
- No third-party/dynamic plugin system. "Plugin" here = the existing Bevy-plugin
  per-crate convention; this is documentation, not a new extensibility API.
- No replacement of the existing hand-written architecture prose
  (`docs/architecture/*.md`). The API reference is **complementary**: prose explains
  concepts, the reference enumerates the surface.

## Constraints & resolved decisions

| Fork | Decision |
| --- | --- |
| Doc depth | Full API reference (items, signatures, module trees) |
| Crate scope | All 19 workspace crates under `crates/*` |
| Authoring scope | Full public-API authoring across all crates (Part B) |
| Generation mechanics | Commit generated model; `make docs`; website stays light; CI freshness check |
| Generator placement | `vmux_docs` — standalone crate **excluded** from the workspace (mirrors `website`) |
| Model representation | Our own stable serde schema (decoupled from `rustdoc-types` `FORMAT_VERSION`) |

Hard constraints discovered:

- **rustdoc JSON is nightly-only** (`-Z unstable-options --output-format json`). The
  repo toolchain stays pinned at stable `1.95.0` for all normal builds; doc-gen alone
  uses a separately pinned nightly.
- **Doc-gen compiles the crates** (bevy/CEF → very large builds). Therefore doc-gen
  runs on demand (`make docs`) and in main CI only — never in the website-deploy job.
- The `website/` crate is **excluded from the main workspace** and has its own
  `Cargo.lock`; its deploy job builds only the website WASM.

## Architecture overview

```
crates/*  (rustdoc //! /// in source)
   │  cargo +nightly rustdoc --output-format json   (per crate)
   ▼
target/doc/<crate>.json   (rustdoc-types, unstable schema)
   │  vmux_docs: parse (rustdoc-types) → translate → filter
   ▼
docs/api/<crate>.json + docs/api/index.json   (OUR stable schema, committed)
   │  copied into website/public/api/  (build step)
   ▼
website renderer (Dioxus): /docs/api, /docs/api/:crate, /docs/api/:crate/:path*
   reuses markdown::Markdown (doc bodies) + syntect (signatures) + DocsLayout
```

The only contract the website depends on is **our** schema. `rustdoc-types`'
instability is absorbed entirely inside `vmux_docs`.

# Part A — Pipeline + UI

## A1. Generator: `vmux_docs`

A standalone Rust binary crate at repo root `vmux_docs/`, **excluded** from the
workspace (add `"vmux_docs"` to root `Cargo.toml` `[workspace] exclude`, exactly like
`website`). It has its own `Cargo.lock`. This honors the "no new workspace crates for
a feature" rule — it is tooling, not a shipped crate.

Dependencies:

- `rustdoc-json` — drives `cargo +<nightly> rustdoc … --output-format json` for a
  given crate manifest and returns the JSON path.
- `rustdoc-types` — typed deserialization of that JSON. Its version is matched to the
  pinned nightly's `FORMAT_VERSION`.
- `serde`, `serde_json`, `anyhow`, `clap` (CLI: which crates, output dir).

Behavior:

1. For each of the 19 crates (a hardcoded list, or discovered from
   `crates/*/Cargo.toml`):
   - Invoke `rustdoc-json` with an explicit toolchain `nightly-YYYY-MM-DD` and
     `--document-private-items=false`, `--no-deps`.
   - Deserialize with `rustdoc-types`.
2. Walk the item tree from the crate root module. Keep **public** items; drop private
   items, blanket/auto trait impls, and `#[doc(hidden)]`.
3. Translate each kept item into our schema (A2), carrying its doc-comment body
   verbatim as Markdown, and a rendered signature string.
4. Resolve intra-doc links (`[Foo]`, `[`Bar`]`) via rustdoc item IDs into `ItemRef`s
   where the target is in-scope; otherwise emit plain text. (Cross-linking is a
   refinement — render as plain text first, wire links second.)
5. Write `docs/api/<crate>.json` per crate and `docs/api/index.json` (crate list with
   versions + the nightly stamp used).

The pinned nightly must be installed locally / in CI
(`rustup toolchain install nightly-YYYY-MM-DD`). The exact date is chosen at
implementation time as the latest nightly whose `rustdoc-types` `FORMAT_VERSION`
matches a released `rustdoc-types`.

## A2. Model schema (stable, serde)

Defined once in `vmux_docs` and mirrored (or shared via a tiny serde-only module) by
the website. JSON on disk. Illustrative shape:

```rust
struct ApiIndex {
    generated_with: String,        // "nightly-YYYY-MM-DD"
    crates: Vec<CrateMeta>,
}
struct CrateMeta { name: String, version: String, blurb_md: String }

struct CrateDoc { name: String, version: String, root: Module }

struct Module {
    path: String,                  // "vmux_history" / "vmux_history::store"
    docs_md: String,               // from //!
    items: Vec<Item>,
    submodules: Vec<Module>,
}

struct Item {
    kind: ItemKind,                // Struct | Enum | Trait | Fn | TypeAlias | Const | Macro | …
    name: String,
    path: String,                  // fully-qualified within crate
    signature: String,             // rendered Rust signature, syntect-highlighted at render
    docs_md: String,               // from ///
    members: Vec<Member>,          // fields / variants / methods / assoc items
    links: Vec<ItemRef>,           // resolved intra-doc links
}
```

`docs_md` is fed straight into the website's existing `markdown::Markdown` component,
so doc comments get the exact vmux styling (headings, code fences, tables, links).

## A3. Committed artifacts

- `docs/api/*.json` are **committed** (not gitignored). They are the build input for
  the website and the thing the CI freshness check guards.
- They are copied into the website's static output at build time (see A5/A6).

## A4. Website renderer

New module `website/src/api.rs` (+ `website/src/api/` directory if it grows; no
`mod.rs`). Components and routes:

Extend the `Route` enum in `website/src/main.rs`:

```rust
#[layout(DocsLayout)]
    #[route("/docs")]                         DocsIndex {},
    #[route("/docs/:slug")]                   DocPage { slug: String },
    #[route("/docs/api")]                     ApiIndex {},
    #[route("/docs/api/:crate_name")]         ApiCrate { crate_name: String },
    #[route("/docs/api/:crate_name/:..path")] ApiItem { crate_name: String, path: Vec<String> },
```

- `ApiIndex` — grid of crates from `index.json` (name + `blurb_md`).
- `ApiCrate` — crate root: `//!` body + module tree + top-level item list.
- `ApiItem` — module or item page: highlighted signature block, rendered `///` body,
  and a members table (fields/variants/methods), each linking onward.

Reuse:

- **Doc bodies:** `markdown::Markdown { content: docs_md }`.
- **Signatures:** expose the currently-private `highlight_html(lang, code)` in
  `markdown.rs` as `pub fn highlight_code(lang, code) -> String`, call it with
  `"rust"` for signature blocks. (Or render signatures as a fenced ```rust block via
  `markdown::Markdown` — decide in impl; exposing the helper is cleaner.)
- **Shell/nav:** `DocsLayout`, theme tokens (`text-text`, `text-text-muted`,
  `text-accent`, `bg-surface`, `border-border`, `bg-code-bg`), and the existing
  scroll-spy. The sidebar gains an "API Reference" entry under `/docs`; while inside
  `/docs/api/*`, the sidebar shows a collapsible crate → module → item tree built
  from the loaded model.

Data loading (Dioxus fullstack SSG):

- Server side (prerender): a `#[server]` fn reads `docs/api/<crate>.json` from disk.
- Client side (hydration / SPA nav): fetch the same JSON from `/api/<crate>.json`
  via `gloo-net` (already a dep). Use `use_server_future` so the same component works
  in both phases.

## A5. SSG / bundle strategy

- Model JSON is served as **static assets**, copied to
  `website/public/api/*.json`. It is **not** `include_str!`'d into the WASM — embedding
  the full API of 19 crates would bloat the binary.
- `static_routes()` in `main.rs` is extended to enumerate prerendered routes:
  - Always: `/docs/api` and `/docs/api/<crate>` for every crate.
  - **Item pages:** for M0, render item detail **client-side** (do not enumerate
    every item in `static_routes`) to bound the prerendered page count. Full per-item
    prerendering (better SEO, more static files) is a later toggle once the item count
    is known and SSG build time measured.
- The website build copies `docs/api/` → its publish dir alongside `style.css`.

## A6. Makefile + CI

Add to `Makefile`:

```
api-docs:        ## regenerate committed API model from rustdoc
	cd vmux_docs && cargo run --release -- --out ../docs/api
```

(Use a name that won't collide with the existing `docs/` dir conventions; `api-docs`
or `docs-api`.)

Website build steps (`website` / `build-website-release`) gain a copy of
`docs/api/*.json` into `website/public/api/` before `dx build`.

CI:

- **Freshness check** (new job, path-gated on `crates/**`, `vmux_docs/**`,
  `docs/api/**`): install the pinned nightly, run the generator, then
  `git diff --exit-code docs/api/`. Fails if the committed model is stale.
- To control cost, run this in / after the existing main build job so the target dir
  is warm (deps already compiled); rustdoc then does less work. CEF build cost is
  acknowledged; this is why doc-gen never runs in the light website-deploy workflow.
- The **website-deploy workflow is unchanged** except for the static copy in A5.

## A7. Seed docs (ships with M0)

So the reference isn't empty on day one, M0 writes, for **every** crate:

- A crate-level `//!` summary at the crate root (`lib.rs`).
- A `///` on each `*Plugin` struct describing the capability it wires up.

Item-level `///` across the full surface is Part B.

# Part B — Full public-API authoring

The content pass. One crate at a time; each crate is an independent unit of work with
its own commit/PR, regenerate, and visual review.

## B1. Doc-comment conventions

- Every public item (`pub` struct/enum/trait/fn/type/const/macro and public
  fields/variants/methods) gets a `///`.
- First line = one-sentence summary (rustdoc convention; also used in lists).
- Bodies are Markdown (they render through `markdown::Markdown`): use fenced
  ```rust examples, tables, and intra-doc links (`[OtherType]`).
- Crate roots get a `//!` overview; non-trivial modules get a `//!`.
- No plain `//` code comments are added (project rule: code has no comments; doc
  comments are the exception and are the whole point here).

## B2. Definition of done (per crate)

- All public items documented (no rustdoc "missing docs" once `#![warn(missing_docs)]`
  is enabled for that crate — enable it crate-by-crate as each is completed).
- `make api-docs` regenerated; `docs/api/<crate>.json` committed.
- Page rendered and visually reviewed at `/docs/api/<crate>` locally.

## B3. Crate ordering (public-facing importance)

1. `vmux_core` → 2. `vmux_browser` → 3. `vmux_terminal` → 4. `vmux_agent` →
5. `vmux_editor` → 6. `vmux_layout` → 7. `vmux_git` → 8. `vmux_space` →
9. `vmux_history` → 10. `vmux_command` → 11. `vmux_service` → 12. `vmux_setting` →
13. `vmux_mcp` → 14. `vmux_team` → 15. `vmux_ui` → 16. `vmux_server` →
17. `vmux_desktop` → 18. `vmux_cli` → 19. `vmux_macro`.

## B4. Tracking

Each crate = a checklist entry (Linear issue or plan task). Part B does not block
Part A shipping; the reference improves as crates are completed.

## Testing strategy

- **Generator unit tests:** a tiny fixture crate with known items
  (`vmux_docs/tests/fixture/`); assert the emitted model (kinds, paths, signatures,
  that private/hidden items are dropped). Snapshot the fixture's JSON.
- **Schema round-trip:** serde serialize/deserialize equality tests.
- **Renderer:** Dioxus component tests rendering an `ApiItem` from a fixture model
  (signature present, doc body rendered, members listed).
- **CI staleness:** the `git diff --exit-code docs/api/` check (A6).

## Risks & mitigations

- **Nightly format churn** — isolated to `vmux_docs` (pinned nightly + matched
  `rustdoc-types`). Website never sees it.
- **Doc-gen build cost (CEF/bevy)** — kept out of the website CI; main-CI check is
  path-gated and runs on a warm target dir.
- **SSG page count for items** — bounded by client-side item rendering in M0;
  full prerender deferred until counts/build-time are measured.
- **Model size in repo** — per-crate JSON files; only changed crates churn per commit.
- **Stale committed model** — guarded by the CI freshness check.

## Rejected alternatives

- **Render raw `rustdoc-types` in the website** — couples the site to the unstable
  JSON schema; raw JSON is bulky and noisy.
- **rustdoc JSON → Markdown → reuse the Markdown pipeline only** — Markdown can't
  express item trees, signature cross-links, or structured nav; loses the "real
  reference" requirement.
- **Reskin `cargo doc` HTML with CSS** — fragile, won't look like vmux, contradicts
  the ask.

## Milestones

- **M0 — Pipeline + UI (Part A):** `vmux_docs`, schema, renderer + routes, Makefile
  target, CI freshness check, seed docs (A7). Ships a working (sparse) reference.
- **M1…M19 — Per-crate authoring (Part B):** one crate per unit, in the B3 order.

M0 is the first implementation plan. Part B crates follow as separate cycles.
